# Monitoring, Alerting, and SIEM Runbook

This runbook covers the local Docker full-stack deployment used for Bestar
pilot and production rehearsal. It does not add a hosted monitoring product.
Local IT can connect these scripts to cron, systemd timers, Windows Task
Scheduler, Splunk forwarders, Filebeat, Graylog, or another SIEM collector.

## What Is Covered

- API structured JSON logs for application startup, HTTP requests, and API
  exceptions.
- Backup freshness alert for PostgreSQL and storage backups.
- Disk usage warning and critical alerts.

## Public Named-Tunnel Pilot Signals

When PUBLIC-DEPLOY-02 is activated, add site-owned checks for:

- `cloudflared` container health/restart count and connector status;
- a public synthetic request that confirms the hostname and Access gate are
  reachable without bypassing Access or storing credentials in the probe;
- Cloudflare Access allow/deny/MFA failures and emergency revocation review;
- Bestar login/refresh/rate-limit/CSRF failures from `auth_audit_events`;
- nginx 413/429/5xx rates and confirmation that authenticated/private
  responses remain `Cache-Control: no-store`;
- local API/database/Redis/queue/document-generation health;
- disk thresholds and matched PostgreSQL plus `storage/` backup freshness.

Correlate an outage with the official Cloudflare status page before changing
the local stack. A public synthetic failure plus healthy LAN/API/database
signals indicates an edge/tunnel/Internet problem; unhealthy LAN signals
indicate an origin problem. Record external alert routing only after a real test
notification is observed. Do not claim an alert platform is connected from
repository configuration alone.
- JSONL audit exports for scan, correction, generated-file, import, and wage
  audit streams.

## API Structured Logs

The API writes one JSON object per line to stdout/stderr.

Important fields:

| Field | Meaning |
| --- | --- |
| `timestamp` | UTC ISO timestamp from the API process. |
| `service` | Defaults to `bestar-api`; can be overridden with `SERVICE_NAME`. |
| `level` | `info`, `warn`, `error`, `debug`, or `verbose`. |
| `context` | Nest or local logging context. |
| `message` | Human-readable message or event summary. |
| `details.event` | Machine-readable event name, for example `http_request` or `api_exception`. |
| `details.requestId` | `x-request-id`, generated by the API when the caller does not provide one. |
| `details.statusCode` | HTTP status code for request logs. |
| `details.durationMs` | Request duration in milliseconds. |

Sensitive keys containing `authorization`, `cookie`, `jwt`, `password`,
`secret`, or `token` are redacted before log output.

View recent API logs:

```bash
docker compose -f infra/docker/compose.local.yml logs --tail=200 api
```

Follow API logs:

```bash
docker compose -f infra/docker/compose.local.yml logs -f api
```

## Backup Freshness Alert

Create backups first:

```bash
scripts/backup-postgres.sh
scripts/backup-storage.sh
```

Check that the newest PostgreSQL and storage backups are non-empty and within
the allowed age window:

```bash
scripts/check-backups.sh
```

Default behavior:

- `BACKUP_DIR=./backups`
- `MAX_BACKUP_AGE_HOURS=25`
- Exits `0` when both backup types are fresh.
- Exits `1` when a backup is missing, empty, or stale.
- Prints JSON lines with event `backup_freshness_check`.

Optional webhook alert:

```bash
ALERT_WEBHOOK_URL='https://example.local/bestar-alerts' scripts/check-backups.sh
```

Cron example:

```cron
15 2 * * * cd /opt/bestar/stripSystem && scripts/backup-postgres.sh && scripts/backup-storage.sh
30 2 * * * cd /opt/bestar/stripSystem && scripts/check-backups.sh >> logs/monitoring.log 2>&1
```

Production expectation:

- Run PostgreSQL and storage backups daily.
- Keep at least one recent backup off the application host.
- Treat stale backup alerts as production-blocking before pilot start.

## Disk Usage Alert

Run:

```bash
scripts/check-disk-usage.sh
```

Default behavior:

- Checks repo root, `storage/`, and `backups/`.
- `DISK_USAGE_WARN_PERCENT=80`
- `DISK_USAGE_CRITICAL_PERCENT=90`
- Exits `0` when all paths are below warning threshold.
- Exits `1` when any path is at warning threshold.
- Exits `2` when any path is at critical threshold or missing.
- Prints JSON lines with event `disk_usage_check`.

Example with stricter pilot thresholds:

```bash
DISK_USAGE_WARN_PERCENT=70 DISK_USAGE_CRITICAL_PERCENT=85 scripts/check-disk-usage.sh
```

Optional webhook alert:

```bash
ALERT_WEBHOOK_URL='https://example.local/bestar-alerts' scripts/check-disk-usage.sh
```

Cron example:

```cron
*/30 * * * * cd /opt/bestar/stripSystem && scripts/check-disk-usage.sh >> logs/monitoring.log 2>&1
```

Response guide:

| Status | Action |
| --- | --- |
| `warning` | Check backup retention and storage growth before next shift. |
| `critical` | Stop nonessential imports, copy backups off-host, and free disk before continuing production work. |

Do not delete original uploads, generated reports, labels, audit exports, or
database volumes during an incident unless a backup and restore point has been
confirmed.

## SIEM Audit Export

Export recent audit streams as JSONL files:

```bash
scripts/export-siem-audit.sh
```

Default behavior:

- Reads the Docker PostgreSQL service through `infra/docker/compose.local.yml`.
- Writes to `audit-exports/`.
- Uses `SIEM_SINCE_INTERVAL="24 hours"`.
- Creates one JSONL file per stream:
  - `pallet-events`
  - `correction-feedback`
  - `generated-files`
  - `import-files`
  - `wage-files`
  - `unloading-wage`

Export a shorter window:

```bash
SIEM_SINCE_INTERVAL='2 hours' scripts/export-siem-audit.sh
```

Export to a collector pickup directory:

```bash
OUTPUT_DIR=/var/log/bestar-audit scripts/export-siem-audit.sh
```

Recommended schedule:

```cron
*/15 * * * * cd /opt/bestar/stripSystem && SIEM_SINCE_INTERVAL='20 minutes' OUTPUT_DIR=/var/log/bestar-audit scripts/export-siem-audit.sh >> logs/siem-export.log 2>&1
```

Collector rules:

- Treat each line as one JSON event.
- Preserve source file name as the SIEM index/source stream.
- Do not mutate `id`, `operatorId`, `correctedById`, `generatedById`, or
  `fileSha256`; they are join keys for investigations.
- Restrict access to `audit-exports/` because it contains user emails, file
  names, scan payloads, and operational metadata.

## Pilot Readiness Checklist

Before pilot:

- `scripts/healthcheck.sh` passes.
- `scripts/backup-postgres.sh` and `scripts/backup-storage.sh` both produce
  non-empty files.
- `scripts/check-backups.sh` exits `0`.
- `scripts/check-disk-usage.sh` exits `0`.
- `scripts/export-siem-audit.sh` creates JSONL files without SQL errors.
- Local IT knows where Docker logs, backup logs, monitoring logs, and SIEM
  export files are stored.

During pilot:

- Check disk every 30 minutes.
- Check backup freshness daily after scheduled backups.
- Export SIEM audit JSONL at least every 15 minutes, or configure the collector
  to read Docker JSON logs and `audit-exports/` directly.

After an incident:

1. Preserve Docker logs and the latest SIEM export files.
2. Preserve PostgreSQL and storage backups.
3. Record affected container numbers, load job numbers, user emails, and time
   window.
4. Search SIEM streams by `requestId`, `fileSha256`, `containerId`,
   `palletRecordId`, `loadJobId`, and user id fields.

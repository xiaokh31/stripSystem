核心判断

长耗时不是“Agent 停着不动”，也不是 Jest 或依赖安装主导。主要成本是：任务要求的全量质量门禁、实现过程中的 Docker/E2E 重建与重试、成倍扩张的视觉矩阵和逐图检查，以及长上下文在大量工具调用之间被模型反复处理。 Docker 缓存已经解决无源码变化时的依赖重装，但源码变化仍会触发应用 build。

时间主要花在哪里

1. 重复 build 与 E2E 调试循环

最近 6 个 Web 运行中，原始 command 字符串至少有 75 组包含 Compose build 或 up --build，有 113 组包含 E2E container、Playwright 或 test:e2e。这些是命令组次数，不是最后报告中的通过用例数。

2. 视觉矩阵和人工逐图检查

WEB-OPS-05 单独生成并检查 236 张 PNG。第 1 轮完成自动化并检查 100 张；第 2 轮继续检查 136 张，再更新 Task/索引/报告和执行清理。第 2 轮本身耗时 25 分 21 秒。

3. 模型上下文被反复处理

最近 6 个 Web 任务累计 284.0M input tokens，其中 273.4M 命中 cache；仍产生 468,778 output tokens 和 145,172 reasoning tokens。一个 supervisor turn 可以包含 174 次命令，并不等于一次模型推理。

4. 连接重试与重复运行

最近 6 个 Web 任务记录 15 个 transport reconnect/fallback 事件，WEB-OPS-05 的有效运行占 13 个；此前还有 一次运行执行 24 个命令、持续 12 分 30 秒后未写终态，随后从新 supervisor run 重新接管。

5. 依赖缓存不是当前主问题

DOCKER-CACHE-01 实测无源码变化 warm build 为 5.88 秒、服务 recreate 为 15.07 秒。临时源码变化 probe 为 114.51 秒，说明依赖层已缓存，但 Next/API/E2E 的应用源码和 build layer 仍会因代码修改重跑。

6. 任务本身是“实现 + 完整关闭”

任务不只写代码，还要求 lint、typecheck、unit、API E2E、production build、Chromium、RBAC、双语、主题、多个 viewport/zoom、真实 fixture、数据清理、截图检查、Task/索引/完成度报告更新。这些都计入同一个 wall time。

哪些是精确数据，哪些只能推断

可以精确复核 Task/run 开始与终态时间、commands/file changes 数、exit code、transport errors、turn 数、token usage、最终结果中列出的 tests 和 screenshot 数量。
当前无法精确分摊 每条命令到底运行多少秒、模型每次推理等待多少秒、人工查看每张图多少秒。现有 events JSONL 没有事件时间戳或 command duration，supervisor 只记录 turn 起止。
有证据的推断 反复出现的 build/E2E command groups、最终任务报告中的显式测试时长、逐图检查点和高 token 累积共同证明，验证重试、 视觉检查与模型工具循环是主要耗时来源。
模型配置边界 审计时全局配置为 gpt-5.6-sol + model_reasoning_effort = "high"。run artifact 没有保存 当时的 model/effort，因此不能把历史运行全部精确归因于 xhigh；但高推理模型会放大长上下文和大量工具回合的时间。

针对性改进

  1. 新需求拆为 WEB-OPS-06/07/08 三个垂直任务，再用 WEB-OPS-09 一次关闭。每个功能任务先 focused test，源码稳定后只做 一次 full suite/build，避免每个小改动重复全栈重建。
  2. WEB-OPS-09 使用受影响页面的 pairwise locale/theme/viewport/role 组合，最终截图建议不超过 36 张；自动几何、DOM、 i18n 和行为断言继续完整，不再用 236 张笛卡尔积截图替代测试。
  3. 最终任务结果除 `1/1 passed` 外，必须报告 wall time、build-bearing command groups、E2E attempts 和重试根因，直接暴露过程成本。
  4. 保持现有 Docker dependency cache。无源码变化复用镜像；源码变化后先在运行容器做 focused checks，再在最终阶段重建 production/E2E image。
  5. 常规 UI、i18n 和已有模式扩展使用 Sol high 即可；xhigh 只用于跨模块架构、难复现并发或复杂生成物诊断。切换前应先记录任务基线， 否则无法证明模型调整带来的收益。
  6. 后续应给 supervisor events 增加 ISO timestamp、command start/end/duration、category、build cache result、E2E attempt 和 screenshot review count。 在此之前，任何“某类耗时占 37%”之类的精确比例都没有数据基础。
